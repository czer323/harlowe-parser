# Harlowe Tokenizer and Parser — Revised Implementation Plan

**Goal:**  
Develop a robust, type-safe, and test-driven Harlowe tokenizer and parser using TypeScript and Chevrotain, with a focus on real-world coverage, rapid feedback, and maintainability. All work should be driven by real `.twee` examples and automated reporting to maximize correctness and minimize manual review.

---

## 1. Guiding Principles

- **Type-Driven Development:** Define and evolve TypeScript types for tokens, AST, and parser results before or alongside grammar rules.
- **Example-Driven Grammar:** Use real Harlowe `.twee` files and macro examples to drive grammar and lexer/parser improvements.
- **Incremental, Test-First:** Add tests and reference examples for every new construct or rule.
- **Automated Feedback:** Batch-parse all examples, aggregate results, and report failures for rapid iteration.
- **Documentation as You Go:** Document tokens, rules, and error cases as they are added.

---

## 2. Implementation Steps

### 1. **Foundation: Types and Utilities**

- **Files:**
  - types.ts — TypeScript interfaces/types for tokens, AST nodes, parser results.
  - printCST.ts — Utility for pretty-printing CST/AST for debugging.
  - `src/utilities/fileUtils.ts` — File discovery and reading helpers.

- **Functions:**
  - `defineTokenTypes()`, `defineASTTypes()`
  - `printCST(node, indent)`
  - `findExampleFiles(dir)`, `readFile(path)`

---

### 2. **Lexer and Parser Core**

- **Files:**
  - lexer.ts — Chevrotain lexer definition for Harlowe.
  - parser.ts — Chevrotain parser for Harlowe grammar.

- **Functions:**
  - `createHarloweLexer()`
  - `createHarloweParser()`
  - `parse(input: string): ParseResult`

- **Approach:**
  - Start with basic constructs (macros, variables, plain text).
  - Expand grammar incrementally using real `.twee` examples.
  - Update types as new constructs are added.

---

### 3. **Automated Batch Parsing and Reporting Tool**

- **Files:**
  - `scripts/batchParse.ts` — Main entry for batch parsing and reporting.
  - `scripts/parseUtils.ts` — Parsing/tokenization helpers.
  - `scripts/reportUtils.ts` — Aggregation and reporting logic.
  - `scripts/types.ts` — Shared types for parse results, errors, etc.

- **Functions:**
  - `main()`
  - `parseAllFiles(inputDir): ParseResult[]`
  - `parseExample(input): ParseResult`
  - `aggregateResults(results): AggregatedReport`
  - `writeJsonReport(report, path)`
  - `writeMarkdownReport(report, path)`
  - `writeFailureFiles(failures, dir)`

- **Goals:**
  - Parse all macro/example files in a directory.
  - Classify each as pass/fail, aggregate errors.
  - Output summary tables and detailed failure reports.
  - Use results to drive grammar/lexer improvements.

---

### 4. **Testing and Fuzzing**

- **Files:**
  - lexer.test.ts — vitest tests for lexer.
  - parser.test.ts — vitest tests for parser.
  - `tests/fuzzing.test.ts` — Grammar-based fuzzing tests (using fast-check).

- **Functions:**
  - `testTokenization()`
  - `testParsing()`
  - `fuzzGrammar()`

- **Approach:**
  - Add/expand tests for every new token/rule.
  - Use batch tool to auto-generate regression tests for each macro/example.
  - Integrate basic fuzzing early to catch edge cases.

---

### 5. **Error Handling and Reporting**

- **Files:**
  - `src/errorProvider.ts` — Custom Chevrotain error provider.
  - `tests/errorProvider.test.ts` — Tests for error messages and recovery.

- **Functions:**
  - `formatError(error): string`
  - `testErrorCases()`

- **Approach:**
  - Prototype error reporting early.
  - Add error cases to batch tool and tests.

---

### 6. **Documentation and Integration**

- **Files:**
  - README.md, `CONTRIBUTING.md`, `CHANGELOG.md`
  - `scripts/generateDocs.ts` (optional) — Auto-generate docs from types/rules.

- **Approach:**
  - Document tokens, rules, and error cases as they are added.
  - Keep changelog and contribution guidelines up to date.

---

## 3. Workflow Summary

1. **Define/Update Types** → 2. **Expand Lexer/Parser** → 3. **Test with Real Examples**  
2. **Batch Parse and Report** → 5. **Fix Failures, Iterate**  
3. **Add/Update Tests and Docs Continuously**

---

## 4. Key Goals

- **Rapid, automated feedback on parser coverage and failures.**
- **Type-safe, maintainable codebase.**
- **Comprehensive, real-world test coverage.**
- **Minimal manual review—focus on fixing what the tool reports.**
- **Documentation and error handling integrated from the start.**

---

This plan will ensure you build a robust, maintainable Harlowe parser with maximum real-world coverage and minimal wasted effort.
